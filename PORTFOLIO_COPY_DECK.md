# Portfolio Copy Deck

Recruiter-minded copy strategy and rewrite guidance for positioning the portfolio as data-engineering-first, workflow-systems-first, and reproducibility-first.

## 1. Diagnosis

### Site-wide positioning

- The current browser title, hero label, and repeated identity lines lead with `Astrophysics`, `Scientific Computing`, and `Cloud Infrastructure`, which makes the site feel domain-first before the engineering value is visible.
- The navbar role line frames the profile as a `Computer Science student`, which weakens first impressions for recruiters.
- The current direction copy reads broad and interest-led instead of role-aligned and execution-led.

### Hero

- The old hero headline reads like a category list rather than a clear statement of what is actually built.
- The subheadline emphasizes research and infrastructure systems without making data workflows, ingestion, validation, or reproducibility immediately obvious.
- A recruiter can understand the domain quickly, but not the transferable engineering value quickly enough.

### Introduction

- The old intro starts from identity and interests instead of starting from systems built and execution style.
- The wording sounds like an early student narrative rather than an early-career builder narrative.
- Engineering rigor is present, but not foregrounded.

### Working Model

- The old section reads like a note about how the site is wired rather than a deliberate public-portfolio curation system.
- It explains mechanics, but not judgment, traceability, or presentation intent.

### Selected Projects

- The section intro talks about curation, but not enough about validated outputs, reusable artifacts, or workflow structure.
- Card summaries are technically real, but still domain-led in several places.
- The category labels are too tied to domain framing.

### Project Detail Pages

- The current summaries and narrative blocks are solid, but still assume too much domain tolerance from non-specialist recruiters.
- Challenge sections sometimes describe scientific questions more than workflow or data problems.
- Repository snapshot lines can inherit raw GitHub descriptions that are less polished than the rest of the site.

### Ongoing Work

- This is the most domain-heavy section.
- Titles and summaries still risk making the work feel like astronomy exploration rather than active workflow engineering.
- The section needs clearer emphasis on ingestion, metadata, structured outputs, validation, and public data platform design.

### Technical Focus

- The existing grouping is domain-first.
- It makes the portfolio easier to understand as an academic or science profile than as an engineering profile.

### Footer and repeated copy

- The footer is functional, but it repeats structural notes rather than reinforcing identity.
- It can sound cleaner, more mature, and more aligned with a data/workflow narrative.

## 2. Main Positioning Gaps

- Domain shows up before engineering value.
- Research framing shows up before workflow design.
- Interests show up before execution.
- Tools are present, but system patterns are not emphasized enough.
- Ongoing work looks exploratory when it should look like active workflow engineering.
- The site needs stronger separation between application domain and core technical identity.

## 8. Project Detail Page Rewrites

### Rubin Sampling

- Short summary: `A reproducible Python pipeline for ingesting public survey light curves, standardizing them into parquet-based artifacts, and evaluating period recovery against Gaia DR3 truth data.`
- Challenge: `Public time-series workflows are easy to prototype and hard to trust once live providers, irregular cadence, and aliasing start to break assumptions. The core problem here was building a pipeline that could ingest unstable external data, standardize it cleanly, and expose evaluation results in a defensible way.`
- Implementation: `The repository turns that problem into explicit stages: Gaia truth-set download, live ZTF ingest with provider fallback and resumable execution, schema-standardized parquet outputs, baseline Lomb-Scargle search, and evaluation tables that surface recovery, ingest status, and known alias failures.`
- Result: `The result is a rerunnable baseline workflow with tracked artifacts at each stage: truth tables, ingest summaries, standardized light-curve batches, period results, evaluation tables, and figure bundles. It reads like a workflow system rather than a one-off notebook experiment.`
- Improved stack label: `Python, Astropy, Pandas, NumPy, Parquet, Gaia DR3, ZTF`
- Improved repository snapshot: `Public baseline pipeline for live ingest, standardized time-series outputs, and evaluation under Rubin-like cadence constraints.`

### T CrB Photometry and Raw-Image Pipeline

- Short summary: `A reproducible Python workflow for ingesting, cleaning, binning, validating, and packaging multi-source photometry plus supporting raw-image assets.`
- Challenge: `Historical and modern observational data tends to fragment across CSVs, notebooks, figures, and archive lookups, which makes reruns and validation hard to trust. The challenge was to keep cleaned products, overlap checks, and raw-image context connected inside a single workflow.`
- Implementation: `I split the project into explicit lanes for modern V analysis, cross-cycle Vis context, and raw-image support. The pipeline produces cleaned parquet outputs, smoothing and binned products, ASAS-SN overlap validation, notebook wrappers, manifests, and archive-aware image staging across PS1, Legacy Survey, and DASCH.`
- Result: `The result is a reusable data workflow with traceable outputs instead of scattered analysis files: standardized tables, exported metrics, figures, and provenance notes all live in one structure. The repository can be rerun, inspected, and extended without losing the context behind each artifact.`
- Improved stack label: `Python, Pandas, NumPy, Matplotlib, Jupyter, Archive Integrations`
- Improved repository snapshot: `Reproducible photometry and raw-image workflow with standardized outputs, overlap validation, and archive-aware provenance.`

### RUWE Radial KS Clustering

- Short summary: `A reusable Python workflow for clustering, membership inference, quality inspection, and cross-dataset comparison on Gaia-based tables.`
- Challenge: `Exploratory clustering analysis often stays trapped in notebooks, making reruns, cross-dataset comparison, and public reuse harder than they should be. The workflow problem was to make intermediate assumptions and quality checks visible without losing flexibility.`
- Implementation: `The repository reorganizes the original analysis into explicit stages: table standardization, Monte Carlo plus KMeans voting for membership inference, CMD generation, RUWE-based quality views, and radial KS testing. A small CLI handles the main paths while archived notebooks preserve provenance.`
- Result: `The result is a cleaner research software workflow that can be rerun across multiple clusters with shared commands, generated figures, and summarized outputs. It preserves transparency while reducing the fragility of manual notebook execution.`
- Improved stack label: `Python, NumPy, Pandas, scikit-learn, Matplotlib, CLI + Notebook Workflow`
- Improved repository snapshot: `CLI-backed clustering and data-quality workflow for Gaia-derived tables, with preserved notebook provenance and multi-cluster outputs.`

## 10. Technical Focus / Focus Areas

### 5 alternative label sets

1. `Data Engineering / Workflow Systems / Research Software`
   Appeal: recruiters for data engineering, research engineering, and data-heavy backend work.

2. `Data Pipelines / Reproducible Systems / Scientific Data Platforms`
   Appeal: recruiters who want clearer data-system language while still tolerating domain context.

3. `Workflow Engineering / Structured Data Systems / Technical Tooling`
   Appeal: backend and platform-oriented recruiters who care about system design more than domain labels.

4. `Data Infrastructure / Pipeline Design / Research Software`
   Appeal: recruiters leaning slightly more infrastructure/platform while still open to research-domain work.

5. `Reproducible Engineering / Data Workflows / Domain-Aware Tooling`
   Appeal: recruiters who value transferable engineering rigor but do not want a generic full-stack narrative.

### Best set

#### Data Engineering

`Pipelines for ingestion, schema shaping, metadata handling, validation, and structured outputs built from messy public datasets.`

#### Workflow Systems

`Raw-to-derived pipeline design, provenance-aware outputs, reproducible reruns, and staged processing that stays inspectable as projects grow.`

#### Research Software

`Domain-specific tools and analysis systems built with engineering discipline, where the scientific context supports the workflow instead of replacing it.`

### Why this grouping is better

This set leads with transferable engineering strengths first and only then acknowledges the domain context. It tells recruiters that the scientific data is real, but the value is in the systems thinking, pipeline design, and reproducibility habits that transfer well beyond astronomy.
